## 一、Critical 级别问题（16项，必须立即修复）

### A1. 路径解析逻辑在5个文件中完全重复
- **来源**: 架构审查 + 代码质量审查
- **位置**: `core/config.py`, `core/logging.py`, `converter/cache.py`, `converter/ocr_models.py`, `converter/ocr_converter.py`
- **描述**: `_resolve_project_root()`, `_is_portable_mode()`, `_resolve_app_dir()` 三个函数在 **5个不同文件** 中完全重复定义。违反DRY原则，修改逻辑需要同步5处。
- **建议**: 提取到 `core/path_utils.py` 作为单一权威来源。

### A2. AppConfig 过度膨胀（God Object）
- **来源**: 架构审查
- **位置**: `core/config.py:151-237`
- **描述**: `AppConfig` 包含50+个字段，混合了LLM配置、Anki配置、OCR配置、UI配置、性能统计、观测指标、任务历史等至少8个不同关注点。
- **建议**: 拆分为 `LLMConfig`, `AnkiConfig`, `OCRConfig`, `UIConfig`, `PerformanceStats`, `ObservabilityConfig` 子模型，通过组合构建。

### A3. workers.py 运行时类型覆盖破坏类型安全
- **来源**: 架构审查
- **位置**: `ui/workers.py:30-34`
- **描述**: `DocumentConverter = None`, `ApkgExporter = None` 等在模块级别将类型名覆盖为 `None`，破坏类型检查器推断。`CardGenerator` 懒加载代理使用 `*args, **kwargs` 完全丧失类型安全。
- **建议**: 使用 `TYPE_CHECKING` 块配合工厂函数替代运行时覆盖。

### A4. OCR全局单例线程安全问题
- **来源**: 性能审查
- **位置**: `converter/ocr_converter.py:51-58`
- **描述**: `_ocr_instance` 是全局单例，多个线程可能同时持有同一个实例并调用 `predict()`，PaddleOCR模型内部可能存在竞态条件。
- **建议**: 为 `predict()` 调用添加独立锁，或使用线程池模式为每个线程维护独立OCR实例。

### A5. LLM调用全局串行锁使并发完全失效
- **来源**: 架构审查 + 性能审查
- **位置**: `card_gen/llm_client.py:91,216`
- **描述**: `self._call_lock = threading.Lock()` 串行化所有LLM调用。即使 `BatchGenerateWorker` 使用 `ThreadPoolExecutor` 并发提交任务，所有LLM调用实际上仍然是串行的，完全抵消了并发设计的意义。
- **建议**: 移除 `_call_lock`（OpenAI SDK的httpx客户端本身线程安全），或为每个并发任务创建独立的 `LLMClient` 实例。

### A6. Anki Gateway推送未使用批量API
- **来源**: 性能审查
- **位置**: `anki_gateway/gateway.py:530-553`
- **描述**: `push()` 方法逐张卡片调用 `_push_single()`，每张卡片都发起独立的AnkiConnect HTTP请求。100张卡片 = 100+次网络往返。`addNotes` 批量API已定义但从未使用。
- **建议**: 使用 `addNotes` 批量添加，`findNotes` 批量查询重复项，将N次网络调用减少到2-3次。

### A7. PDF OCR逐页同步渲染阻塞
- **来源**: 性能审查
- **位置**: `converter/ocr_converter.py:1609-1629`
- **描述**: PDF OCR处理逐页串行执行：渲染图片 → OCR识别 → 下一张。即使GPU可用也无法利用并行。
- **建议**: 实现生产者-消费者模式预渲染下一页，或使用 `ThreadPoolExecutor` 并行处理多页。

### A8. 资源泄漏风险 — OCR云转换器
- **来源**: 代码质量审查
- **位置**: `converter/ocr_converter.py:796`
- **描述**: `httpx.Client` 在 `with transport as client:` 块内创建，但如果 `_convert_via_cloud` 的 `try` 块抛出异常前就失败，`transport` 不会被释放。
- **建议**: 将 `transport` 的创建移入 `with` 语句，或使用 `contextlib.ExitStack`。

### A9. 线程安全问题 — 配置模块级初始化
- **来源**: 代码质量审查
- **位置**: `core/config.py:88-93`
- **描述**: `_CONFIG_CACHE` 字典在模块加载时被初始化为可变对象，`CONFIG_PATH` 等模块级常量在初始化时就调用 `_resolve_app_dir()`，多线程环境下可能导致竞态条件。
- **建议**: 使用 `threading.Lock` 保护模块级初始化或使用懒加载。

### A10. UI线程安全问题 — 跨线程UI更新
- **来源**: UI/UX审查
- **位置**: `ui/workers.py` 多处
- **描述**: Worker在 `run()` 方法中直接通过 `self._config` 访问主线程配置对象，存在竞态条件。
- **建议**: 在Worker初始化时将需要的配置值深拷贝为本地属性。

### A11. 崩溃处理中硬编码中文
- **来源**: UI/UX审查
- **位置**: `ui/app.py:311`
- **描述**: 崩溃对话框文本硬编码为中文，英文用户看到中文错误信息。
- **建议**: 根据 `config.language` 动态选择语言。

### A12. 版本管理三重真相源
- **来源**: 构建CI审查
- **位置**: `pyproject.toml:7`, `src/ankismart/__init__.py:1`, `packaging/ankismart.iss:4`
- **描述**: 版本号在三个地方独立维护，没有任何机制保证三者同步。
- **建议**: 使用单一真相源，推荐在 `pyproject.toml` 中定义版本，通过 `importlib.metadata.version()` 运行时读取。

### A13. 关闭事件中的异常吞没
- **来源**: UI/UX审查
- **位置**: `ui/main_window.py:713-724`
- **描述**: `_shutdown_pages` 中捕获所有异常并仅记录debug日志，可能导致Worker未正确停止即退出，引发段错误。
- **建议**: 对关键资源（如QThread）使用 `wait(timeout)` 确保线程终止。

### A14. API文档完全缺失
- **来源**: 文档审查
- **位置**: `docs/api.md` 不存在
- **描述**: `contributing.md` 多次引用 `api.md`，但该文件不存在。所有公共API（`IConverter`, `ICardGenerator`, `IAnkiGateway`, `CardDraft`, `GenerateRequest` 等）均无参考文档。
- **建议**: 创建完整的 `docs/api.md`。

### A15. 用户手册缺失
- **来源**: 文档审查
- **位置**: `docs/user-guide.md` 不存在
- **描述**: 项目没有独立的用户手册，`docs/examples/README.md` 引用了不存在的 `../user-guide.md`。
- **建议**: 创建包含安装、首次使用引导、各功能页面操作指南的用户手册。

### A16. 配置模块承担过多职责
- **来源**: 架构审查
- **位置**: `core/config.py`（618行）
- **描述**: 该文件不仅负责配置加载/保存，还包含加密逻辑、迁移逻辑、备份管理、使用量统计、任务历史、观测指标记录等。
- **建议**: 拆分为 `config_loader.py`, `config_migrator.py`, `config_backup.py`, `usage_tracker.py`。

## 二、High 级别问题（61项，重点修复）

### 代码结构与架构

#### H1. converter模块循环导入风险
- **位置**: `converter/converter.py:6-11`
- **描述**: 顶层导入多个子转换器，OCR converter使用延迟导入表明已意识到此问题但未系统性解决。
- **建议**: 使用依赖注入或注册表模式。

#### H2. UI层直接依赖底层实现细节
- **位置**: `ui/workers.py:828-901`, `ui/preview_page.py:694-850`
- **描述**: UI层直接构建 `DocumentConverter` 并传递10+个配置参数，直接实例化 `LLMClient` 和 `CardGenerator`。
- **建议**: 引入工厂类或依赖注入容器。

#### H3. 两套独立错误处理系统
- **位置**: `core/errors.py` vs `ui/error_handler.py`
- **描述**: 两者功能重叠但互不关联，`ErrorHandler` 重新做字符串模式匹配而非基于 `ErrorCode`。
- **建议**: 统一为单一错误处理框架。

#### H4. gateway.py内嵌750行JavaScript/CSS模板
- **位置**: `anki_gateway/gateway.py:30-380`
- **描述**: 300+行JavaScript字符串加上多个HTML模板字符串，不应混在Python代码中。
- **建议**: 提取到 `anki_gateway/templates/` 独立资源文件。

#### H5. workers.py文件过大（1352+行）
- **位置**: `ui/workers.py`
- **描述**: 包含7个Worker类、质量检查函数、重试逻辑、图片合并逻辑等。
- **建议**: 拆分为 `convert_worker.py`, `generate_worker.py`, `push_worker.py`, `export_worker.py`, `connection_workers.py`。

#### H6. 方法复杂度过高
- **位置**: `card_gen/generator.py:296-469`（generate方法170+行）、`generator.py:150-294`（_split_markdown 145行）、`ui/error_handler.py:368-533`（classify_error 166行）
- **建议**: 拆分为多个私有方法，降低圈复杂度。

#### H7. 重复的安全函数
- **位置**: `_is_disallowed_remote_ip` 在 `apkg_exporter.py:140-148` 和 `ocr_converter.py:422-430` 重复；`_structure_error_message` 在 `apkg_exporter.py:344-347` 和 `validator.py:87-90` 重复
- **建议**: 提取到 `core/security.py`。

#### H8. 重复的下载限制函数
- **位置**: `converter/ocr_converter.py:499-596`
- **描述**: `_download_cloud_text_with_limit` 和 `_download_cloud_bytes_with_limit` 逻辑几乎完全相同。
- **建议**: 合并为泛型函数或使用 `typing.overload`。

#### H9. LLMClient `__del__` 可能吞掉异常
- **位置**: `card_gen/llm_client.py:123-139`
- **描述**: `__del__` 中捕获所有异常，如果 `close()` 持有锁可能导致死锁。
- **建议**: 移除 `__del__`，依赖显式 `close()` 和 context manager。

#### H10. 使用反射检查函数签名
- **位置**: `card_gen/generator.py:75-108`
- **描述**: 每次LLM调用都使用 `inspect.signature()` 检查 `chat` 方法是否支持 `timeout` 参数。
- **建议**: 在 `CardGenerator.__init__` 中一次性检查并缓存结果。

### 安全问题

#### H19. 加密密钥派生强度不足
- **位置**: `core/crypto.py:56-59`
- **描述**: 使用 `platform.node()` + `platform.machine()` + 固定salt派生Fernet密钥，这些值可预测。
- **建议**: 使用CSPRNG生成主密钥，或使用PBKDF2/Argon2从用户密码派生。

#### H20. DPAPI加密回退到机器绑定密钥
- **位置**: `core/crypto.py:146-153`
- **描述**: 无主密钥时自动回退到可预测的机器标识符，便携模式下加密强度不足。
- **建议**: 强制要求用户设置主密钥，或首次启动时生成并提示保存随机密钥。

#### H21. 无应用层用户认证
- **位置**: 全局
- **描述**: 任何能访问该计算机的用户都可以使用已配置的API密钥和AnkiConnect连接。
- **建议**: 考虑添加可选的应用锁（PIN或密码）。

#### H22. LLM提示词注入风险
- **位置**: `card_gen/generator.py:296-449`
- **描述**: 用户文档内容直接作为user_prompt传递给LLM，没有任何清洗或分隔。
- **建议**: 使用明确的分隔符（如 `<document>` 标签），系统提示词中强调不要遵循文档中的指令。

#### H23. 日志可能泄露敏感信息
- **位置**: `card_gen/llm_client.py:260-268`, `core/logging.py:113-119`
- **描述**: 异常消息可能包含API key片段；JsonFormatter将所有extra字段写入日志。
- **建议**: 记录错误前过滤敏感信息，增加敏感字段过滤器。

#### H24. 云OCR端点验证不足
- **位置**: `converter/ocr_converter.py:118-124`
- **描述**: 允许使用HTTP端点，API密钥将明文传输。
- **建议**: 拒绝非HTTPS的云OCR端点配置（localhost除外）。

#### H25. LLM API请求无TLS验证配置
- **位置**: `card_gen/llm_client.py:78-85`
- **描述**: 自定义base_url使用HTTP时API密钥明文传输。
- **建议**: 对非localhost的base_url强制要求HTTPS，并在设置界面中警告。

### 性能问题

#### H35. 转换缓存无淘汰策略
- **位置**: `converter/cache.py:46,123-139`
- **描述**: 缓存文件无限增长，没有TTL、LRU或大小限制。
- **建议**: 添加LRU淘汰策略（如最大500MB或1000个条目）。

#### H36. 文件哈希计算每次都读取全文
- **位置**: `converter/cache.py:53-59,76`
- **描述**: 对于200MB PDF，即使缓存命中也需要先完整读取文件来计算哈希。
- **建议**: 添加快速路径：先检查 `mtime + size` 的轻量哈希。

#### H37. 图片合并时所有图片同时加载到内存
- **位置**: `ui/workers.py:975-984`
- **描述**: 50张高分辨率图片可能导致数百MB内存占用。
- **建议**: 流式处理：逐张加载、转换、立即释放。

#### H38. Markdown/Text编码检测加载全文件到内存
- **位置**: `converter/markdown_converter.py:31`, `text_converter.py:58`
- **描述**: `file_path.read_bytes()` 一次性读取整个文件。
- **建议**: 使用chardet的流式检测器，只读取前10-50KB。

#### H39. 语义去重使用O(n²)比较
- **位置**: `ui/workers.py:173-179`
- **描述**: 对每张新卡片遍历所有已接受的问题，使用SequenceMatcher进行O(n*m)比较。
- **建议**: 使用MinHash/LSH或TF-IDF余弦相似度近似。

#### H40. BatchConvertWorker重复构建Converter
- **位置**: `ui/workers.py:1060`
- **描述**: 每次重试都创建新的 `DocumentConverter` 实例。
- **建议**: 在 `run()` 开始时构建一次，整个批量转换过程中复用。

#### H41. LLM客户端无连接池复用
- **位置**: `card_gen/llm_client.py:79-84`
- **描述**: 无代理时使用OpenAI SDK默认客户端，每次请求可能重新建立TCP连接。
- **建议**: 始终创建并使用httpx.Client连接池。

#### H42. docx转换中重复查找表格元素
- **位置**: `converter/docx_converter.py:134-135`
- **描述**: 对每个表格元素都遍历 `doc.tables` 列表，O(tables²)复杂度。
- **建议**: 预先构建 `{element: table}` 的字典映射。

#### H43. `_is_meaningful_text` 低效字符串遍历
- **位置**: `converter/ocr_pdf.py:102-107`
- **描述**: 创建新字符串再遍历计数，O(n)内存 + O(n)时间冗余。
- **建议**: 使用单次遍历计数。

#### H44. Cloud OCR轮询使用阻塞sleep
- **位置**: `converter/ocr_converter.py:894`
- **描述**: `time.sleep()` 期间无法响应取消请求。
- **建议**: 使用 `self._cancel_event.wait(poll_interval)` 替换。

#### H45. 配置保存时全量序列化
- **位置**: `core/config.py:415-446`
- **描述**: 每次调用都执行全量序列化 + YAML写入。
- **建议**: 添加防抖机制，将多次保存合并为一次。

#### H46. PDF渲染默认300 DPI过高
- **位置**: `converter/ocr_pdf.py:66-70`
- **描述**: 对于OCR来说150-200 DPI通常足够，4.17x缩放导致内存和时间浪费。
- **建议**: 将默认值降至 `200/72 ≈ 2.78`。
